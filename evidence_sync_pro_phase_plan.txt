PHASE 3: CORRELATION ENGINE (Week 5-6) [CHECKPOINTS 3A, 3B]
────────────────────────────────────────────────────────────

OBJECTIVES:
  ✓ Build core correlation rules engine
  ✓ Implement 5 core correlation rules
  ✓ Add time skew detection
  ✓ Implement explainable matching logic

TASKS:

  3A: CORRELATION ENGINE CORE - Week 5, Day 1-2
      □ Build correlation_engine.py
        • Load all parsed events into in-memory graph
        • Define Correlation result class:
          {source_artifact_1, source_artifact_2, match_type, 
           confidence_score, explanation}
        • Implement rule execution pipeline:
          1. Load rules from YAML
          2. For each rule, find matching event pairs
          3. Calculate confidence score
          4. Store result with full explanation
        • Add caching to avoid recomputation
        
      □ Build event_normalizer.py
        • Convert all event timestamps to UTC
        • Detect & store timezone info per artifact
        • Flag events with missing timestamp
        
      □ Build time_sync_detector.py
        • Detect clock skew between devices
        • Flag impossible timestamps (future dates)
        • Calculate time offset per device
        
      CHECKPOINT 3A: Correlation engine loads rules, executes 1 simple rule

  3B: CORE CORRELATION RULES - Week 5-6, Day 3-5
      □ IP Correlation (ip_correlation.py)
        Rule: Match same IP across EVTX + cloud logs + browser history
        Confidence factors:
          • Exact IP match: +50 points
          • Same user on both sources: +25 points
          • Timestamps within 1 minute: +15 points
          • Different devices: +10 points (proves cross-device activity)
        Manipulability: IPs can be spoofed (medium difficulty)
        
      □ User Identity Correlation (user_identity_correlation.py)
        Rule: Match username/email across Windows + cloud + mobile
        Factors:
          • Exact username match: +40 points
          • Similar email domain: +15 points
          • Timestamp within 5 minutes: +30 points
        Manipulability: Hard to fake without account compromise
        
      □ Timestamp Correlation (timestamp_correlation.py)
        Rule: Find events from same user on different devices at same time
        Factors:
          • Same timestamp (±2 min tolerance): +35 points
          • Accounts for clock skew: check time_sync_detector output
          • User identity match: +30 points
        Manipulability: Timestamps can be faked (easy)
        
      □ File Hash Correlation (file_hash_correlation.py)
        Rule: Match file hashes across NTFS + cloud storage
        Factors:
          • Exact SHA-256 match: +60 points
          • Same modification time: +20 points
          • Proves file exfiltration to cloud
        Manipulability: Hard (cryptographic)
        
      □ Geolocation Correlation (geolocation_correlation.py)
        Rule: Detect impossible travel between locations
        Factors:
          • iPhone location data (from iCloud backup)
          • WiFi SSID from EVTX (network connection logs)
          • Windows location history (if enabled)
        Logic:
          • Extract lat/long from each source
          • Calculate travel distance between locations
          • If distance > 300km in < 1 hour → FLAG as impossible
          • Confidence: 0% if physically impossible, 100% if consistent
        Manipulability: Very hard (GPS spoofing is difficult)
        
      □ Unit tests (5+ per rule):
        • Match detection works
        • Confidence calculation correct
        • Edge cases handled (missing fields, malformed data)
        
      CHECKPOINT 3B: 5 correlation rules working, 25+ tests passing

ESTIMATED HOURS: 50-60 hours
TIME ALLOCATION: 4-5 hours/day × 2.5 weeks

PHASE 4: CONFIDENCE SCORING (Week 7) [CHECKPOINT 4]
────────────────────────────────────────────────────

OBJECTIVES:
  ✓ Build artifact manipulability database
  ✓ Implement hybrid confidence scoring (ML + heuristic)
  ✓ Add explainability layer
  ✓ Train base model on forensic datasets

TASKS:

  4.1 BUILD MANIPULABILITY DATABASE - Week 7, Day 1
      □ Create manipulability_db.json
        Each artifact gets:
          • Score (0-100): how easy is it to fake?
          • Evidence: academic references/proof
          • Detection: how to detect tampering?
          
        Example entries:
        {
          "EVTX_logs": {
            "manipulability_score": 35,  # Hard to fake entirely, but can be cleared
            "evidence": "Event log tampering is detectable via:
                        1. Gap detection (missing EventIDs)
                        2. Timestamp consistency check
                        3. Reverse engineering log structure",
            "notes": "Can be cleared but leaves forensic trace"
          },
          "Windows_Registry": {
            "manipulability_score": 40,
            "notes": "Last write times can be modified with tools"
          },
          "Prefetch": {
            "manipulability_score": 50,
            "notes": "Can be faked with prefetch injection attacks"
          },
          "File_Hash_SHA256": {
            "manipulability_score": 90,  # Very hard (cryptographic)
            "notes": "Collision found but impractical to exploit"
          },
          "iPhone_Location": {
            "manipulability_score": 70,
            "notes": "GPS spoofing possible but requires sophisticated setup"
          }
        }
        
  4.2 CONFIDENCE SCORING SYSTEM - Week 7, Day 2-3
      □ Build confidence_scorer.py
        Final confidence = Base_Score × Artifact_Reliability
        
        Base_Score: Calculated by correlation rule
        Artifact_Reliability = 1 - (manipulability_score / 100)
        
        Example:
          IP match rule gives: 100 points (perfect match)
          EVTX has manipulability 35 → reliability = 0.65
          Windows cloud logs have manipulability 45 → reliability = 0.55
          Combined artifact reliability = (0.65 + 0.55) / 2 = 0.60
          Final confidence = 100 × 0.60 = 60%
        
      □ Build explainability layer:
        For each correlation, output:
        {
          "correlation": "IP_match",
          "devices": ["Windows_Server", "OneDrive"],
          "confidence_score": 62,
          "breakdown": {
            "base_score": 100,
            "artifact_weights": [
              {"artifact": "EVTX", "manipulability": 35, "weight": 0.60},
              {"artifact": "OneDrive_logs", "manipulability": 45, "weight": 0.60}
            ]
          },
          "explanation": "IP 192.168.1.100 found in both EVTX (2024-01-15 14:23:00)
                         and OneDrive sync log (2024-01-15 14:24:15). 
                         Timestamp alignment within 1 minute suggests same session.
                         Confidence reduced to 62% due to EVTX manipulability (35%)."
        }
        
  4.3 ML CONFIDENCE MODEL (OPTIONAL v1.0) - Week 7, Day 4-5
      □ Train confidence_model.pkl
        Dataset: forensic case studies (find academic datasets)
        Features:
          • artifact_types (one-hot encoded)
          • correlation_rule_type
          • timestamp_alignment_seconds
          • user_identity_match (boolean)
          
        Target: binary classification (reliable=1, unreliable=0)
        
        OR: Simple heuristic-only scoring (no ML) for MVP
        
      □ Unit tests:
        • Confidence calculation deterministic & repeatable
        • Explainability text is accurate
        • Scores range 0-100%

TIPS FOR SUCCESS:
─────────────────

1. Windows Development Only:
   • Test each parser on real Windows artifacts
   • Never assume Linux tools are available
   • Validate all dependencies install via pip

2. Forensic Accuracy First:
   • Every timestamp must be correct (timezones!)
   • Every correlation must be explainable
   • Assume this will be used in court

3. Incremental Testing:
   • Don't wait until Phase 7 to test end-to-end
   • Test integration every 2 weeks
   • Get sample real-world data ASAP

4. Beta Testing is Critical:
   • Don't assume your design is intuitive
   • Watch users use the tool (screen recording)
   • Most feedback will come from UI, not logic

5. Documentation as You Go:
   • Write docstrings while coding
   • Document assumptions in code comments
   • Create examples as you build features

6. Git Discipline:
   • Commit after each completed task
   • Use meaningful commit messages
   • Create branches for experimental features

The key is consistent daily progress & quick feedback loops.
